## Build Sequence Overview

TiltBreak is a single Xcode project with two targets: the main app and the WidgetKit extension. There is no traditional backend/frontend split — Johnny owns the Domain and Data layers (entities, repository protocols, repository implementations, use cases, services, and infrastructure), and Jett owns the Presentation layer (Views, ViewModels, animations, and the widget UI). Both engineers work in the same repo.

**The sequence is dictated by two hard constraints:**

First, the widget target must exist from day 1 — it cannot be bolted on later because TiltBreakApp.swift's deep link routing depends on it at compile time. B1 is a hard prerequisite for the entire project.

Second, the tilt interrupt is the product's core mechanic and the riskiest user-facing feature. It now has two phases: Phase 1 (RageChannel bubble tap game) and Phase 2 (90-second guided countdown). The BubbleEngine rendering performance is the only genuine unknown in the entire build — it must be validated on a real device in sprint 1 before any other TiltInterrupt UI work proceeds. If it fails, there is a one-day fallback. This gets built first.

**Phase 1 — Foundation (Sequential, Johnny leads):**
B1 (Project Setup) must complete before anything else. B2 (Entities + Schema — now includes TapEvent) and B3 (Repository Protocols — now includes TapEventRepository) follow immediately. Once B3 is done, Jett can begin coding ViewModels against protocol interfaces even before implementations exist. B6 (WallClockTimer — Phase 2 only) and B18 (BubbleEngine + TapDecayDetector) should both be built in this phase. BubbleEngine must be profiled before F4 integration begins.

**Phase 2 — Core mechanic (Parallel):**
Johnny builds B4 (TiltEvent + TapEvent + Session repositories), B11 (tilt interrupt use cases: Start, RecordTap, TransitionToPhase2, Complete, Abandon), and B5 (UserProfile repository). Jett builds F1 (App Entry + Deep Link), F2 (Onboarding), and F4 (TiltInterrupt — full two-phase implementation) in parallel. The full two-phase tilt interrupt must reach working state before other features.

**Phase 3 — Data + Composure layer (Parallel):**
Johnny builds B7 (ComposureRepository), B8 (SyncRepository), B12 (composure + streak use cases), B13 (post-session use cases), B14 (onboarding use cases). Jett builds F3 (HomeView), F5 (InterruptCompleteView), F6 (PostSessionReviewView). These can run simultaneously.

**Phase 4 — Payment layer (Sequential then Parallel):**
B10 (StoreKit 2 infrastructure) must complete before B15 (payment use cases), F9 (PaywallView), and F7 (ComposureDashboard blur gating). B10 is the riskiest task and must start as early as possible. App Store Connect configuration (HUMAN TASK) is a hard prerequisite for B10 sandbox testing — Bartholomew must action this before Phase 4.

**Phase 5 — Remaining features + Widget UI:**
B9 (Notifications), B16 (Export use case), B17 (Settings use case), F8 (ComposureCardView), F10 (SettingsView), F11 (Widget UI), F12 (Animation polish). These are largely independent and can be distributed freely across both engineers once Phase 3 is stable.

**Parallel work possible:** Yes. Jett can begin F1 and F2 immediately after B1 and B3 complete. From Phase 2 onward, Johnny and Jett should be working simultaneously on most tasks, with explicit integration sync points at B6→F4 (Phase 2 timer), B18→F4 (Phase 1 engine), B11→F4 (use cases), B12→F3/F5, B13→F6, B10→F7/F9, B16+F8→F7. F4 depends on B18 completing its performance validation checkpoint before Jett integrates BubbleEngine into RageChannelView.

## Code Standards Checklist

Standards that apply to TiltBreak beyond portfolio defaults:

- [ ] MVVM + Clean Architecture enforced across all features
- [ ] No business logic in Views — Views are dumb, all logic in ViewModels and UseCases
- [ ] No UIKit imports in Domain or Data layers (UIKit permitted only in Presentation layer and Core/Sharing/)
- [ ] Repository protocols in Domain/Repositories/, implementations in Data/Repositories/
- [ ] Dependency injection — no singletons except EntitlementManager (which is injected into PaymentRepositoryImpl, not accessed globally)
- [ ] WallClockTimer is the source of truth for Phase 2 elapsed time ONLY — Timer.publish is UI animation only. Phase 1 elapsed time is computed from UserDefaults "phase1StartDate" Date() diff in use cases and ViewModels.
- [ ] BubbleEngine built (B18) and profiled on a real device before F4 Phase 1 integration begins — this is a mandatory performance checkpoint, not optional
- [ ] TapEvent records written for ALL users from day 1 — no entitlement check in TapEventRepositoryImpl or RecordTapEventUseCase
- [ ] Phase 1 and Phase 2 each have their own UserDefaults wall-clock key: "phase1StartDate" and "phase2StartDate". AbandonTiltInterruptUseCase must clear BOTH regardless of which phase is active. On app launch, if no active TiltEvent exists in SwiftData, both keys must be cleared.
- [ ] No missed-bubble counter, no score display, no fail state in RageChannelView — Phase 1 is an outlet, not a test. Any UI element that grades or scores Phase 1 behavior is a design violation.
- [ ] ComposureRepository writes data for ALL users — no entitlement check at data write layer. Any tier check at data write is an architecture bug.
- [ ] Entitlement check (CheckEntitlementUseCase) called on every scene phase .active transition in HomeViewModel
- [ ] Tilt button haptic + ring-expand animation fire BEFORE StartTiltInterruptUseCase call (optimistic UI — non-negotiable)
- [ ] Post-session review: bottom sheet over HomeView, not navigation push
- [ ] StoreKit 2 product IDs: Config.swift only, never hardcoded at call sites
- [ ] Every purchase flow has loading, success, failure, and pending states handled
- [ ] Restore purchases implemented and visible in both Paywall and Settings
- [ ] Subscription status checked on every .active scene phase transition
- [ ] Trial expiry handled with exactly one local notification scheduled at onboarding; cancelled on paid conversion
- [ ] No force unwraps without explicit justification comment
- [ ] No clinical language, no diagnostic terminology, no "therapy/treatment/disorder/anxiety" in any string literal
- [ ] Composure score copy is always progress-framed — never raw tilt count displayed
- [ ] Tilt button minimum tap target: 80×80pt with .frame + .contentShape
- [ ] Widget target exists from day 1 (B1 precondition) — cannot be added late
- [ ] Swift naming: types UpperCamelCase, properties/functions lowerCamelCase
- [ ] One type per file, file named exactly as the type it contains
- [ ] No dead code committed
